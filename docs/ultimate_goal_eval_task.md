## 2. 이 작업의 위치

`html-to-editable-pptx` 프로젝트의 ULTIMATE_GOAL 문서는 T-line 세션에서 사용자와의 8차례 인터뷰를 거쳐 v0.1로 합의되었습니다. 본 작업은 그 합의가 *얼마나 견고한지*를 외부 시각으로 검증하는 것입니다.

본 작업은 **합의를 뒤집는 작업이 아닙니다.** 합의된 사실(ultimate goal 한 문장, (1)(2)(3) 의존성 분석, 5개 commitment, 합의 보존소 모델, 두 독자 모델, YAML 인덱스 등)은 이미 사용자와의 직접 인터뷰로 확정되었습니다. 본 작업의 결과물은 *합의를 흔드는 것*이 아니라, *합의의 **경계**가 정확한지를 외부 시각으로 점검*하여 **다음 인터뷰에 던질 질문 후보**를 생성하는 것입니다.

`ULTIMATE_GOAL_v0.1.md`를 가장 먼저 정독한 뒤, 나머지 문서들을 *그 기준으로 비교*하며 읽는 것을 권장합니다.

## 4. 평가의 본질 — "경계 검증"

ULTIMATE_GOAL 문서는 v0.1에서 다음 4축으로 *내부 점검*을 거쳤습니다:

- **외향 O1**: 이 문서가 다른 문서의 영역을 침범하는가?
- **외향 O2**: 이 문서가 재논의를 어렵게 만드는 권위 표현을 가지는가?
- **내향 I1**: 이 문서가 마땅히 다루어야 할 핵심을 빠뜨리는가?
- **내향 I2**: 이 문서가 다른 문서에 흩어진 핵심을 끌어와야 하는가?

내부 점검에서 *발견된 모든 위반은 이미 수정되었습니다.* 따라서 본 작업은 **내부 점검이 놓쳤을 만한 경계 위반**을 외부 시각에서 찾는 것입니다.

특히 다음 종류의 발견이 가치가 높습니다:

| 유형 | 설명 | 예시 |
|---|---|---|
| 잠재 충돌 | ULTIMATE_GOAL 문서의 어떤 문장이 다른 문서의 어떤 문장과 *암묵적으로 충돌* | "§4-C5는 PPTX 비풍부 약속, 그러나 architecture_v0.1.md는 PPTX 풍부 기능 모듈을 가정" |
| 미명시 위임 | ULTIMATE_GOAL 문서가 *어떤 것을 다른 문서에 위임한다*고 했으나 *그 다른 문서가 실제로는 다루지 않음* | "C2의 operable scope를 GOAL_PROBLEM_v0.1.md에 위임했으나 해당 구체 정의가 없음" |
| 비대칭 추상도 | 같은 commitment 안에서 한 부분은 추상, 다른 부분은 구체로 일관성이 깨짐 | "C1은 'PPT slide'라는 구체 명사, C3는 추상 원칙뿐" |
| 합의 누락 | 인터뷰 기록에는 있으나 문서에 반영되지 않은 합의 | (외부 평가자는 인터뷰 기록 접근 불가 → 이 유형은 평가 어려움) |
| YAML-산문 불일치 | YAML 블록과 그 위 산문이 *세부적으로 어긋남* | 문서명/키/commitment 이름 불일치 |

## 5. 평가 절차 — 권장 순서

### 5.1. ULTIMATE_GOAL_v0.1.md 단독 읽기 (1차 패스)

다른 문서를 보기 전에 ULTIMATE_GOAL_v0.1.md만 정독합니다. 다음 질문에 답해보십시오:

- 이 문서가 정의하는 ultimate goal을 한 문장으로 다시 말할 수 있는가?
- (1)(2)(3) 액션의 의존성 구조를 그림 없이 설명할 수 있는가?
- 5개 commitment의 이름을 외울 수 있는가?
- 이 문서를 *누가 수정할 수 있는가*에 대해 명확한가?
- YAML 블록의 주요 최상위 키를 외울 수 있는가?

이 자기 점검에서 *모호한 부분*이 있다면 그것이 **내향 I1 후보**입니다.

### 5.2. ULTIMATE_GOAL_v0.1.md ↔ 다른 문서 교차 검증 (2차 패스)

다른 6개 문서를 차례로 읽으며, ULTIMATE_GOAL_v0.1.md와의 *모든 교차점*을 기록합니다.

각 교차점에 대해:

```text
교차점 ID: CX-NN
ULTIMATE_GOAL_v0.1.md의 위치: §N L?
다른 문서의 위치: <doc>:L?
관계: [정합 / 충돌 / 위임-실현됨 / 위임-미실현 / 중복]
근거: <한 문장>
인터뷰 질문 후보: <한 문장, 사용자에게 던질 질문>
```

*위임-미실현* 발견이 가장 가치 있습니다 — ULTIMATE_GOAL 문서가 *"다른 문서가 다룬다"*고 했는데 실제로 그곳에 없거나 부분적으로만 있는 경우입니다.

### 5.3. YAML 블록 단독 점검 (3차 패스)

YAML 블록만 *산문과 분리해서* 읽고 다음을 점검합니다:

- YAML이 *합의 보존소* 역할에 충분한가? (Agent가 산문을 안 읽어도 핵심을 잡을 수 있는가)
- YAML 키가 다른 문서의 어떤 키와도 *충돌하지 않는가*?
- YAML이 *과대 단순화*된 부분이 있는가? (산문의 미묘함을 잃은 곳)
- YAML이 *과대 박제*된 부분이 있는가? (자주 바뀔 항목인데 고정된 곳)

YAML은 산문이 권위이고 YAML은 derivative입니다. 따라서 YAML이 산문보다 *덜* 정확한 것은 허용되지만, *틀린* 것은 안 됩니다.

## 7. 비목표 (Non-goals)

### Case: Type

- **합의를 도전하지 마라.** (1)(2)(3) 우선순위, 합의 보존소 모델, 두 독자 모델, YAML 도입 등은 *이미 합의된 사실*입니다.
- **ULTIMATE_GOAL_v0.1.md를 직접 수정하지 마라.** 평가만 작성하고, 수정 제안은 보고서 본문에만 남깁니다.
- **다른 문서도 직접 수정하지 마라.** 본 작업은 *평가 보고서 1개* 작성입니다.

### Case: State

- **모든 발견이 위반이 아니다.** *점검했으나 위반 아님*도 가치 있는 결과입니다.

### Case: Performance: Over

- **3일 이상 쓰지 마라.** 본 평가는 *합의된 문서의 외부 점검*입니다.
- **새 합의를 제안하지 마라.** 발견은 *기존 합의 안에서* 해석합니다.

### Case: Performance: Under

- **답할 수 없는 발견이 있어도 괜찮다.** 외부 평가자는 *인터뷰 기록에 접근할 수 없음*. 어떤 발견은 *외부에서는 판단 불가*로 끝나도 됩니다.

## 8. 검증 규칙

산출물이 다음을 만족해야 완료로 간주합니다:

1. **주요 섹션 모두 점검**되었음이 보고서에서 확인 가능
2. **YAML 블록도 별도 점검**됨
3. **최소 1개 이상의 인터뷰 질문 후보** 제시
4. **위반이 아닌 발견도 명시**
5. **다른 6개 문서와의 교차점이 최소 4개 이상 점검됨**

## 9. 본 작업이 T-line 세션에 어떻게 환류되는가

본 평가 보고서가 완료되면 T-line 세션이 다음 작업을 합니다:

1. *우선순위 인터뷰 질문 후보*를 사용자에게 제시
2. 사용자 답변에 따라 ULTIMATE_GOAL_v0.1.md 추가 갱신 (사용자 동의 필요)
3. v0.2로 commit

따라서 본 보고서의 *가장 중요한 산출물은 §4 우선순위 인터뷰 질문 후보*입니다. 다른 섹션은 *그 질문 후보의 근거*를 제공합니다.
